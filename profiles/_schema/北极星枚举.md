# 北极星指标枚举 + 复盘判定规则（v0.1）

> §8 待决 #4。最硬的一块:复盘逻辑要按 `profile.north_star[].metric` 分支,读不同判定规则。
> onboarding 第1问定商业目标 → 推荐对应北极星 → 用户确认(或数据反推纠偏)。
> 引擎不写死指标,迪迪=「涨粉率+完播率」只是其中一种组合。

---

## 枚举表

| metric | 适用商业目标 | definition | 判定规则（复盘读） | 阈值字段 |
|---|---|---|---|---|
| `涨粉率` | 涨粉 / 立人设 | 涨粉 / 千播 | > target 视为受众匹配;低于=钩子或选题不对 | `target` `direction:max` |
| `完播率` | 通用底线(几乎都看) | 完播人数 / 播放 | 先过 `floor` 合格线,再奔 `target` | `floor` `target` |
| `收藏率` | 干货 / 工具号 | 藏 / 赞 | > target = 受众精准(入场券),但需先有播放 | `target` |
| `GMV/客单` | 带货 | 单条归因 GMV 或客单价 | 单条归因 GMV 达标;看转化非看播放 | `target` |
| `加微率` | 导私域 | 加微/进群人数 / 播放(或评论口令转化) | 评论口令→加微的转化率达标 | `target` |
| `互动率` | 泛流量 / 破圈 | (赞+评+转) / 播放 | 达标但**预警**:易跑偏精准,慎用为主指标 | `target` |

---

## 判定逻辑（引擎伪代码）

```
for ns in profile.north_star:
    v = 从后台取 ns.metric 的实测值
    if ns.floor and v < ns.floor:     标记「未过合格线」→ 归因钩子/选题
    elif v >= ns.target:               标记「达标」→ 该套路加权,沉淀
    else:                              标记「待复看」→ 看发布时长(<48h不判)
```

> 配合记忆:判数据要看时间线——绝对数脱离发布时长无意义,<48h 不判爆/扑。

---

## 组合规则

- 一个号可有多个北极星(迪迪=涨粉率+完播率),`north_star` 是数组,复盘逐个判。
- **主指标 vs 底线**:完播率常作所有号的底线(floor),不一定是主指标。
- `anti_metrics` 明确排除(迪迪排除播放量/总涨粉),复盘报告里不让它们带偏结论。

## onboarding 推荐映射（第1问商业目标 → 默认北极星）

| 用户选的目标 | 推荐 north_star | 可叠加 |
|---|---|---|
| 涨粉/做大 | 涨粉率 | + 完播率 |
| 带货 | GMV/客单 | + 完播率 + 收藏率 |
| 导私域 | 加微率 | + 完播率 |
| 立人设/接广 | 涨粉率 + 互动率 | + 完播率 |
| 做干货/工具 | 收藏率 | + 完播率 + 涨粉率 |

> 推荐≠强制。给账号 URL 时,🔵 数据反推会拿用户真实近 30 条算基线,纠偏嘴说的目标(§3 自动抓)。
